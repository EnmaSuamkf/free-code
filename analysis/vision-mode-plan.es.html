👁 Modo Visión para Free-Code — Investigación y Plan

Darle a free-code ojos (tu pantalla) y oídos (tu voz), para resolver problemas juntos mientras interactúan. «Tengo un navegador abierto — ¿qué es esto?»

Proyecto de referencia estudiado: svpino/alloy-voice-assistant, clonado y analizado en /tmp/alloy-voice-assistant.

1. Qué hace alloy-voice-assistant (y qué nos enseña)

Es un único script de Python de ~170 líneas que demuestra el ciclo completo de interacción de principio a fin:

┌──────────────┐     ┌──────────────┐     ┌───────────────┐     ┌─────────┐
│ Mic escucha  │ ──▶ │ Whisper STT  │ ──▶ │ GPT-4o        │ ──▶ │ TTS     │
│ (2º plano)   │     │ (local)      │     │ texto+imagen  │     │ (voz)   │
└──────────────┘     └──────────────┘     └───────┬───────┘     └─────────┘
                                                  ▲
                                     ┌────────────┴────────────┐
                                     │ Un hilo de webcam guarda│
                                     │ el ÚLTIMO frame en RAM  │
                                     └─────────────────────────┘
  1. Un hilo en segundo plano mantiene el frame más reciente de la webcam en un buffer. No se sube nada.
  2. El micrófono escucha en segundo plano con detección de actividad de voz; cuando dejas de hablar, dispara un callback.
  3. El callback transcribe tu voz con Whisper local, toma el frame actual como JPEG en base64, y envía transcripción + una imagen + historial del chat a un modelo multimodal.
  4. La respuesta se reproduce en voz con TTS en streaming.
La idea clave: no hace falta entender vídeo. El modelo ve una captura tomada en el instante en que preguntas, más el historial de la conversación. Eso hace que la funcionalidad sea barata, rápida y construible con cualquier modelo con visión que free-code ya soporta.

Nota del entorno (esta máquina): Ubuntu GNOME sobre Wayland, sin ninguna CLI de capturas instalada. En Wayland las apps no pueden capturar la pantalla silenciosamente — la captura pasa por el XDG Desktop Portal (Screenshot para capturas puntuales; ScreenCast + PipeWire para un stream persistente tras un único diálogo de permiso). Esto es una ventaja: el sistema operativo nos regala el flujo de consentimiento.

3. Arquitectura propuesta — una extensión por defecto vision.ts

Se distribuye como browse-command.ts: una extensión por defecto que expone /see, /voice y /vision live, con backends intercambiables por plataforma para captura, voz-a-texto y texto-a-voz. Sin cambios en el core — todo fluye por la API de extensiones existente.

Fase 1

/see — captura bajo demanda ⭐ empezar aquí

/see ¿qué es este error? → captura la pantalla → sendUserMessage([texto, imagen]). El redimensionado reutiliza Photon (loadPhoton()) y el pipeline autoResizeImages (2000×2000) en vez de un JPEG ~1568px hardcodeado. Backends elegidos con isWaylandSession(): portal XDG como camino primario en Wayland (no fallback), scrot/import (X11), screencapture (macOS), PowerShell (Windows). Detectar el backend disponible y avisar si falta todo. Ya cumple el ejemplo que motivó la idea, escrito en vez de hablado.

≈ 1–2 días

Fase 2

/voice — entrada pulsar-para-hablar

Un atajo registrado con pi.registerShortcut() graba el micrófono (pw-record / arecord / sox); transcripción con whisper.cpp local (privado, gratis) o API (OpenAI Whisper, Groq, Gemini). La transcripción se vuelca al editor con ctx.ui.setEditorText() para revisar, o se envía directa. Idioma configurable. Detectar dependencias (whisper/tts no instalados aquí).

≈ 2–3 días

Fase 3

/vision live — el ciclo de alloy

Manos libres, modelado como un modo (precedente mode.ts). Escucha en segundo plano con VAD; cada frase toma el frame más reciente (portal ScreenCast → un diálogo de permiso por sesión) + transcripción → sendUserMessage([...], {deliverAs:"steer"}) para inyectar aunque el agente streamee. Respuestas habladas con TTS en streaming e interrumpible (pi.on(…) detecta nueva utterance) y mostradas en la TUI. Estado con ctx.ui.setStatus("vision",…); controles con setWidget().

≈ 4–6 días

Fase 4

variante web-ui — la demo más barata

En packages/web-ui el navegador lo da todo de forma nativa: selector de pantalla con getDisplayMedia(), Web Speech API para STT, speechSynthesis para TTS. Cero dependencias nativas — ideal para prototipar la UX del modo en vivo.

≈ 2–3 días, paralelizable

4. Privacidad, coste, latencia

  • Privacidad: la pantalla muestra gestores de contraseñas, tokens, mensajes privados. Reglas: opt-in explícito por sesión; capturar solo en el momento de la pregunta (nunca subida continua); indicador siempre visible en modo en vivo; configuración vision.exclude; los diálogos del portal en Wayland refuerzan el consentimiento.
  • Coste: un JPEG reducido a 1568px ≈ 1.1–1.6k tokens de entrada; una sesión en vivo de 30 preguntas ≈ 40–50k tokens de imagen. Adjuntar imagen solo cuando la pregunta la necesita (heurística o palabra clave «mira…»).
  • Latencia (objetivo < 5 s, como alloy): parada del VAD ~0.3s + Whisper local base ~0.5–1s + captura ~0.1s + primer token del modelo ~1–2s + inicio del TTS en streaming ~0.5s.

5. Veredicto

Sí — muy factible. alloy-voice-assistant valida el modelo de interacción (captura en el momento de la pregunta, sin necesidad de vídeo), y free-code ya tiene las partes difíciles: un pipeline de mensajes multimodal, modelos con visión, y una API de extensiones que acepta contenido de imagen en los mensajes de usuario. El trabajo genuinamente nuevo son tres adaptadores bien conocidos — captura de pantalla, STT, TTS — más una UX de consentimiento cuidadosa. La Fase 1 (/see) por sí sola ya entrega el «¿qué es esto en mi pantalla?» y se puede construir de inmediato.

Entornos reales del repo: CLI (default-extensions), VS Code (packages/vscode-free-code + free-desktop-host/activate-vscode.mjs), macOS (free-desktop-host/src/stdio-mac.mjs, puente stdio que reusa el core) y web-ui. No existe un paquete FreeCodeMac separado. Los slash commands se registran una sola vez en default-extensions (core compartido); solo los atajos (registerShortcut) y la command palette de VS Code necesitan registro por entorno. Ver el checklist del AGENTS.md (y corregir ahí DEFAULT_APP_KEYBINDINGS, que no existe).

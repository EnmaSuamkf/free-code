👁 Free-Code Vision Mode — Investigation & Plan

Give free-code eyes (your screen) and ears (your voice), so you can solve things together while you interact. “I have a browser open — what is this?”

Reference project studied: svpino/alloy-voice-assistant, cloned and analyzed at /tmp/alloy-voice-assistant.

1. What alloy-voice-assistant does (and teaches us)

It is a single ~170-line Python script that proves the whole interaction loop end-to-end:

┌─────────────┐     ┌──────────────┐     ┌───────────────┐     ┌─────────┐
│ Mic listens │ ──▶ │ Whisper STT  │ ──▶ │ GPT-4o        │ ──▶ │ TTS     │
│ (background)│     │ (local)      │     │ text + image  │     │ (voice) │
└─────────────┘     └──────────────┘     └───────┬───────┘     └─────────┘
                                                 ▲
                                    ┌────────────┴────────────┐
                                    │ Webcam thread keeps the │
                                    │ LATEST frame in memory  │
                                    └─────────────────────────┘
  1. A background thread continuously keeps the freshest webcam frame in a buffer. Nothing is uploaded.
  2. The mic listens in the background with voice-activity detection; when you stop talking it fires a callback.
  3. The callback transcribes your speech with local Whisper, grabs the current frame as base64 JPEG, and sends transcript + one image + chat history to a multimodal model.
  4. The answer is spoken back with streamed TTS.
The key insight: you do not need video understanding. The model sees one snapshot taken at the instant you ask, plus conversation history. That makes the feature cheap, fast, and buildable with any vision model free-code already supports.

Environment note (this machine): Ubuntu GNOME on Wayland, no screenshot CLI installed. On Wayland, apps cannot silently grab the screen — capture goes through the XDG Desktop Portal (Screenshot for snapshots; ScreenCast + PipeWire for a persistent stream after a one-time permission dialog). This is a feature: the OS provides the consent flow for free.

3. Proposed architecture — a vision.ts default extension

Ship it like browse-command.ts: a default extension exposing /see, /voice and /vision live, with pluggable platform backends for capture, speech-to-text, and text-to-speech. No core changes needed — everything flows through the existing extension API.

Phase 1

/see — snapshot on demand ⭐ start here

/see what is this error? → capture screen → sendUserMessage([text, image]). Downscaling reuses Photon (loadPhoton()) and the autoResizeImages pipeline (2000×2000) instead of a hardcoded ~1568px JPEG. Backends chosen with isWaylandSession(): XDG portal as the primary path on Wayland (not a fallback), scrot/import (X11), screencapture (macOS), PowerShell (Windows). Detect the available backend and warn if nothing is present. Already fulfills the motivating example, typed instead of spoken.

≈ 1–2 days

Phase 2

/voice — push-to-talk input

A shortcut registered with pi.registerShortcut() records the mic (pw-record / arecord / sox); transcription via local whisper.cpp (private, free) or API (OpenAI Whisper, Groq, Gemini). Transcript is dropped into the editor with ctx.ui.setEditorText() for review, or sends directly. Language configurable. Detect dependencies (whisper/tts not installed here).

≈ 2–3 days

Phase 3

/vision live — the alloy loop

Hands-free, modeled as a mode (precedent mode.ts). Background VAD listening; each utterance grabs the freshest frame (ScreenCast portal → one permission dialog per session) + transcript → sendUserMessage([...], {deliverAs:"steer"}) to inject even while the agent streams. Answers spoken via streamed, interruptible TTS (pi.on(…) detects a new utterance) and rendered in the TUI. Status via ctx.ui.setStatus("vision",…); controls via setWidget().

≈ 4–6 days

Phase 4

web-ui variant — cheapest demo

In packages/web-ui the browser gives everything natively: getDisplayMedia() screen picker, Web Speech API for STT, speechSynthesis for TTS. Zero native dependencies — ideal place to prototype the live-mode UX.

≈ 2–3 days, parallelizable

4. Privacy, cost, latency

  • Privacy: the screen shows password managers, tokens, private messages. Rules: explicit opt-in per session; capture only at question time (never continuous upload); always-visible indicator in live mode; vision.exclude config; Wayland portal dialogs reinforce consent.
  • Cost: one downscaled 1568px JPEG ≈ 1.1–1.6k input tokens; a 30-question live session ≈ 40–50k image tokens. Only attach an image when the question needs one (heuristic or wake-word “look…”).
  • Latency (target < 5 s, like alloy): VAD stop ~0.3s + local Whisper base ~0.5–1s + capture ~0.1s + model first token ~1–2s + streamed TTS start ~0.5s.

5. Verdict

Yes — very feasible. alloy-voice-assistant validates the interaction model (snapshot-at-question-time, no video needed), and free-code already has the hard parts: a multimodal message pipeline, vision models, and an extension API that accepts image content in user messages. The genuinely new work is three well-understood adapters — screen capture, STT, TTS — plus careful consent UX. Phase 1 (/see) alone already delivers “what is this on my screen?” and can be built immediately.

Real environments in this repo: CLI (default-extensions), VS Code (packages/vscode-free-code + free-desktop-host/activate-vscode.mjs), macOS (free-desktop-host/src/stdio-mac.mjs, an stdio bridge that reuses the core), and web-ui. There is no separate FreeCodeMac package. Slash commands are registered once in default-extensions (shared core); only shortcuts (registerShortcut) and the VS Code command palette need per-environment registration. See the AGENTS.md checklist (and fix there DEFAULT_APP_KEYBINDINGS, which does not exist).

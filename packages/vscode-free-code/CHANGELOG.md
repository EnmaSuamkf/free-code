# Changelog

## [Unreleased]

### Added

- **`@free/free-desktop-host`**: shared Node implementation of the chat host (RPC, tabs, RAG, etc.) bundled into the VSIX via esbuild; also used by the macOS app (`apps/free-code-macos`) over a stdio JSONL bridge.
- **`media/chat.html`**: shared HTML shell for the chat webview (VS Code and macOS `WKWebView`).
- **`media/webview-mac-preamble.js`**: `acquireVsCodeApi` shim and base64 host injection for the macOS app.
- Slash command **`/profile`** in the slash catalog (RPC allowlist + webview fallback): list / use / create / save / delete session presets for tools, skills, agents, and theme.
- Slash command **`/browse`** in chat (same headed `agent_browser` prompt as the globe button): `/browse [url] [goal]`; omitting the URL uses `https://www.google.com`. Advertised in the slash catalog next to `/gemini`.
- RAG slash commands in the plugin chat: `/rag-kb create|delete|use <name>`, `/rag-kb list`, `/rag addFile <path>`, `/rag addGroup <folder>`, `/rag search <query>`, `/rag list`, and `/rag remove <filename>`. New conversations start with no selected KB (no implicit default); users must choose one with `/rag-kb use <name>`. `addGroup` applies the add-file workflow to each supported file in the folder and reports added/skipped/failed counts. Data is scoped under `~/.free-code/knowledgeBase/<kb>/`, and requests hit the local RAG server (`http://localhost:8085` by default, configurable with `FREE_CODE_RAG_SERVER_URL` via `free-code.env`). **`/rag-kb use`** and profile-applied KBs may trigger a follow-up model turn with **`GET /discover`** metadata (`*.knowledge.md`) when the server supports it; **`/rag addFile`** copies one file at a time (optional `<stem>.knowledge.md` sidecar is not required) and accepts **`.txt`**.
- **Export conversation**: toolbar **Export** and command **Free Code: Export conversation to Markdown** write a `.md` file. When the RPC session is on disk, the export matches the CLI session export (thinking, tools, compaction, etc.); otherwise the extension saves the sidebar transcript as a fallback.
- **Inline questionnaire UI**: when the agent invokes the bundled `questionnaire` tool, the chat now renders a single inline card with the question prompt, radio options, an optional "Type something." free-text textarea, and `Cancel` / `Submit` buttons — instead of stealing focus with a Cursor/VS Code QuickPick at the top of the window. Multi-question runs show a tab strip (`■ Q1 / □ Q2 / ✓ Submit`) and remember partial answers when navigating between tabs. Implements the new `extension_ui_request` method `questionnaire` (`coding-agent` ≥ this release); older agents fall back to the previously-shipped chained QuickPick / InputBox path automatically.
- **Working indicator**: a small spinner appears in the status bar while the agent is processing a prompt, so it is always clear that work is in progress.
- **Live agent activity row**: while a turn is running, the chat now keeps a temporary "Agent working" row visible under the latest activity, with the current phase, elapsed time, and time since the last progress event. It updates for reasoning, tool calls, tool results, answer streaming, session startup, and cancellation.
- **Stalled-turn auto-recovery**: if the chat is still waiting, no agent progress events arrive for 60 seconds, and RPC `get_state` reports `isStreaming: false`, the extension sends one hidden `continue` prompt to resume the task and shows a visible recovery notice.
- **Message queue**: the chat input stays enabled while the agent is busy. Pressing Enter/Send while busy queues the message instead of blocking. A "1 Queued" panel appears above the input showing the pending text with an edit button (restore to input) and a cancel button. When the agent finishes, the queued message is sent automatically. Escape while busy now also clears the queue before aborting.
- **Tool call/result persistence**: tool calls and results are now stored in tab history and restored as collapsed `<details>` blocks when switching tabs or reloading the panel, so they are no longer lost on tab switch.
- **Parallel chat execution**: each chat tab now owns its own `free-code --mode rpc` child process, session sync, streaming buffers, abort handling, and stalled-turn watchdog. Starting work in one tab no longer blocks a second tab from running its own conversation at the same time.
- **Subagent widget for RPC**: `/sub <task>` and natural-language `subagent_create` tool calls now surface the bundled `subagent-widget` in the VS Code/Cursor chat. Each subagent gets a live clickable card plus its own tab; clicking the card focuses that tab, and finished subagents render a `[subagent-result]` message with the full result back into the parent conversation like the CLI.
- Subagent tabs now show an expanded activity timeline for the child agent (thinking deltas, text progress, tool start/end events, and process completion), while the parent chat keeps the compact clickable card.
- Subagent cards/tabs now key off the widget's unique execution id instead of the visible `Subagent #N`, so late results from an older subagent cannot overwrite a newer card/tab with the same visible number.

### Changed

- **Session monitor**: when the RPC child is not running or stats are not yet available, the panel shows a short explanation and a **Load session** button instead of all-zero metrics; **Load session** re-requests stats from the host (same warm-up path as opening the panel). The host waits until `get_state` shows a **model**, the **session file matches** the tab when known, and (if the webview already has rows) the RPC **message count is non-zero**, so the grid does not sit on empty zeros while the session hydrates.
- RAG VS Code host: `GET /discover` **404** (server without that route) skips metadata injection without an error prompt; `/rag addFile` no longer requires a `<stem>.knowledge.md` beside each primary.

- Chat input footer uses a **single Composer-style row**: workspace and model pills on the left; icon-only attach / Export / Browser plus circular **Send** (↑) on the right. Long pill labels truncate with ellipsis; pills cap at a sensible max width so one visible pill can use more space when the other is hidden.
- While the agent is processing, the circular control switches from **Send** (↑) to **Stop** (square) with a distinct color; clicking it aborts the turn (same as **Escape**). **Enter** still queues another message when busy.
- Attached files (drop / paste / Attach button chips in the input footer) are no longer rendered as long absolute paths in the user message bubble. The bubble now shows one clickable chip per attachment with just the file basename (full path on hover); clicking the chip opens the file in the editor (`open_file` → `vscode.window.showTextDocument`). The agent still sees the same shell-style absolute path tokens it used to (the host appends them to the prompt sent over RPC), so the `read` / `edit` tools work exactly like in the TUI. History persistence and `abort_undo` (Escape) keep the attachment list, so reloading or aborting a turn re-renders the chips and re-attaches them to the input.
- The embedded agent **no longer passes `--no-extensions` by default**, so MCP and agent extensions load like the terminal CLI and `/pick-tools` can show the same groups as the TUI. Use the new setting **`free-code.noExtensions`** (default `false`) to restore the previous minimal spawn. Changing that setting or **`free-code.executablePath`** restarts the RPC process on the next action.
- Errors for missing RPC commands `get_tool_picker_state` / `set_tool_picker` append a short hint to use a current `free-code` binary and **Executable path**.
- The **Thinking** block now streams **expanded by default** so reasoning tokens are visible live as the model produces them (useful to debug long or failing turns where the final answer never arrives — e.g. `gemini-2.5-pro` taking a long time to answer "que modelo eres?"). When the assistant's final-answer text begins streaming, any still-open thinking blocks for the current turn are collapsed automatically so the chat stays focused on the answer; if the turn errors out before the answer arrives, the block stays open with the partial reasoning the model emitted.
- The **Browser** button in the chat actions row launches Google Chrome via `bash -lc` with `--remote-debugging-port=9222`, a dedicated `--user-data-dir="$HOME/.free-code/chrome-debug-profile"`, and startup flags (`--no-first-run`, `--disable-default-browser-check`). If port 9222 already exposes DevTools, the extension skips spawning a duplicate Chrome. After a new spawn, it waits until `GET /json/version` succeeds, then sends a structured prompt so the agent attaches with `agent_browser` over CDP (`--cdp http://127.0.0.1:9222`, `sessionMode` preferring `auto`) and opens a default landing page before `snapshot -i`. macOS only for now; on other platforms the extension shows a warning instead of launching anything.
- **Browser button is now cross-platform (macOS, Linux, Windows)**: `handleLaunchChromeDebug()` resolves an installed Chrome/Chromium (falling back to Edge/Brave) via `resolveChromeExecutable()` instead of the hardcoded macOS `Google Chrome.app` path, and spawns it directly with an args array (no `bash -lc`) so paths with spaces work everywhere; if no browser is found it explains how to install one or start it manually on `--remote-debugging-port=9222`.
- **Single tab on launch**: Chrome is launched with the landing URL as its start page so the one startup tab is the content, and the CDP attach prompt now `snapshot`s that tab instead of calling `open` (which creates a second tab over CDP and leaves a blank "New Tab" behind).
- **CDP attach prompt** now tells the agent the CDP connection does not persist between calls: include `--cdp http://127.0.0.1:9222` + sessionMode `"fresh"` on every `agent_browser` call, prefer `navigate` over `open` (reuses the tab), and recover a blank/about:blank snapshot with `tab list` / `tab <id>`.

### Fixed

- **Browser (globe) button**: launching Chrome for CDP no longer spawns a second instance on a quick second click, and the extension waits until `http://127.0.0.1:9222/json/version` responds before sending the attach prompt to the agent. That avoids Chrome's intermittent "Something went wrong when opening your profile" dialog (two processes racing on the same `--user-data-dir`) and reduces cases where `agent_browser --cdp` ran before the port was up and fell back to opening another headed browser. Chrome is started with `--no-first-run` and `--disable-default-browser-check`. The injected attach prompt now prefers `sessionMode: "auto"` first (with a one-line retry hint using `fresh` only if an implicit session already exists). The long CDP attach instructions are sent over RPC with **`showInChat: false`**, so they no longer appear as a user message bubble in the sidebar (only a short hint row).
- Switching to a different chat while another tab is streaming no longer renders the `Agent working` row in the inactive chat. The webview still knows the backend is busy for queuing purposes, but only the tab that owns the active turn shows the live working indicator.
- The **Thinking** block no longer disappears after switching to another sidebar view (or another VS Code window) and coming back to the chat. Two bugs combined to lose it: (1) `retainContextWhenHidden: true` was being set on `webview.options`, which is a `WebviewOptions` object that does not accept that field — VS Code ignored it, so hiding the view tore down the webview DOM. The flag now goes through the third argument of `registerWebviewViewProvider` (`{ webviewOptions: { retainContextWhenHidden: true } }`), where it actually keeps the panel alive while hidden. (2) Thinking text was streamed to the webview but never persisted into `tab.messages`, so any full disposal (window reload, extension host restart, tab switch) would re-render the chat from history without the thinking block. The host now buffers the per-block thinking text and `pushHistory("thinking", …)` at the same boundaries it flushes the assistant bubble (`tool_execution_start`, `agent_end`), and `restore_history` rebuilds those blocks collapsed by default — same final state the user saw after the answer streamed.
- Copy / paste / cut / undo / redo / select-all (Cmd/Ctrl + C/V/X/Z/Shift+Z/Y/A) now work again in the chat input. Previously these shortcuts were registered in `package.json` `contributes.keybindings` with `"command": ""` scoped to `webviewId == 'freeCode.chatView'`, intending to "release" the keys for the webview. In VS Code/Cursor an empty-command keybinding does not free the key — it adds a no-op binding that **swallows** the event before the native `<textarea>` inside the webview can handle it, so clipboard and undo/redo silently stopped working. Removed those entries; the underlying Electron browser handles copy/paste/undo natively on the textarea, and `Enter` is still handled by the existing `keydown` listener in `media/chat.js` (Enter submits, Shift+Enter inserts a newline).
- `RPC timeout for command abort` no longer appears as a chat error row when pressing Escape during a slow turn or when sending the first prompt in a newly-created tab while the previous tab is still streaming. The plugin now sends `abort` as a best-effort fire-and-forget signal: the server-side `session.abort()` calls `agent.waitForIdle()` which can take longer than the normal 30 s RPC timeout while a tool / MCP / HTTP cancellation is finishing, so awaiting the response surfaced spurious errors (one per Escape press, each delaying the next action by 30 s). Local cleanup (popping the last user message, clearing the pending assistant bubble) still happens before signalling abort, and `busy: false` is still driven by the agent's `agent_end` event when cancellation actually completes.
- MCP tools in the chat (e.g. `mcp-atlassian`, `mcp-github`): the first call no longer silently fails because damage-control auto-blocked in RPC. The extension now handles the agent's `extension_ui_request` events (`confirm`, `select`, `input`, `editor`, `notify`, `setStatus`, and fire-and-forget UI methods) and shows native VS Code dialogs. The first MCP tool call now shows the same "Allow MCP tools for the rest of this session?" prompt the terminal CLI shows; accepting authorizes MCP for the rest of the session. Requires a `packages/coding-agent` build that includes the matching damage-control fix.
- Paste into the chat textarea after a native dialog (MCP authorization, `showQuickPick`, `showInputBox`): VS Code returns focus to the previous panel (usually the editor) when these dialogs close, which caused the next Cmd+V to land in whatever file was open instead of the chat input. The extension now calls `webviewView.show(false)` and posts a `focus_input` message to the webview after every dialog, so focus returns to the chat textarea.
- When **Executable path** is the default `free-code`, the extension now spawns the workspace’s `packages/coding-agent/dist/cli.js` with `node` if it exists (after `npm run build`), so `/pick-tools` and RPC `get_tool_picker_state` work without relying on a newer global install on PATH.
- `/pick-tools` with no optional MCP/extension groups (common when the agent runs with `--no-extensions`) no longer opens an empty modal; the chat shows the same summary as `/tools` instead. The tool-picker dialog can be closed with **Escape** or a click on the dimmed backdrop, and cancel clears the busy state reliably.
- **Apply** on the tool picker with nothing to change (no optional groups) skips `set_tool_picker` and only shows the tools summary, avoiding `RPC timeout for command set_tool_picker` when the on-PATH `free-code` is older or minimal. The RPC client also accepts legacy error lines that omitted `id`, and the modal uses explicit `display` toggles so it dismisses reliably.

### Added

- Active-workspace indicator in the chat input toolbar mirroring the CLI footer line `~/Documents/repositories/free-code (main)`: shows the workspace folder name and the current git branch, with the home-relative path in the tooltip. Hidden when no workspace folder is open. Clicking the indicator reveals the folder in the OS file explorer (`revealFileInOS`). The label refreshes when VS Code adds/removes a workspace folder and when `.git/HEAD` changes (file watcher mirrors `FooterDataProvider`'s 500 ms debounce so atomic HEAD updates do not flood the webview), so branch switches in the SCM panel update the indicator without reloading the panel.
- `/model <id> [<provider>]` directly switches the active model from the chat input without opening the picker (e.g. `/model gemini-2.5-flash [google-vertex]`). Mirrors the row format the picker shows so the user can "type what they see"; also accepts the pair without brackets, the `<provider>/<id>` form, and just `<id>` when it is unambiguous across configured providers. Validates against the same `get_available_models` list the picker uses, so unknown ids/providers, ambiguous ids, and ids without configured auth all return helpful chat errors with suggestions instead of silently spawning a turn. Bare `/model` keeps opening the existing modal picker.
- Compatible with the new opt-in `set-active-model` extension shipped at `packages/coding-agent/examples/extensions/set-active-model/`. With that extension copied (or symlinked) to `~/.free-code/agent/extensions/`, the LLM gets a `set_active_model` tool it can call when the user asks in natural language ("change the model to gemini-2.5-pro [google-vertex]", "cambia el modelo a claude-sonnet-4-5"); previously the model had no capability to do that and replied "I cannot change the model. My capabilities do not include switching between different models." The plugin now listens for `tool_execution_end` events with `toolName === "set_active_model"` and refreshes the active-model indicator in the input toolbar immediately on success, so the chip reflects the new model without waiting for the next session sync. The change takes effect on the next user turn (the current turn finishes with the previous model). The `/model` slash command and the model picker are unchanged and still work as before.
- `/model` opens a webview model picker mirroring the CLI `/model` selector (RPC `get_available_models` + `set_model`): only models with configured auth are listed, the current model is marked with a ✓ and floats to the top, and a search box filters by provider/id/name. Click a row or hit Enter on the highlighted row to apply; the chat shows a `Model set to <id> [provider] (name)` confirmation. Escape or a click on the dimmed backdrop cancels.
- Active-model indicator next to the attach button in the chat input toolbar: shows the current model id (full `name — id [provider]` in the tooltip) and clicking it opens the `/model` picker without writing `/model` into the chat history. The label refreshes automatically after applying a new model in the picker, when the RPC client first connects, when the webview reloads, and when a tab switch loads its session.
- Tool calls and tool results are now rendered as their own rows in the chat (tool name header + arguments/result body), matching the CLI layout instead of only showing a `Tool: <name>` status line. The current assistant bubble is closed when a tool starts executing, so any text the model emits after a tool result lands in a new bubble **below** the tool rows (CLI order: preamble → tool call → tool result → final answer) instead of being merged into the preamble bubble above the tool activity. Long tool results are truncated with a "N more chars hidden" footer to keep the chat readable.
- Thinking / reasoning tokens are streamed into a collapsed `Thinking` block (`<details>`) inside the chat, not just the bottom status line. The block stays collapsed by default to keep the conversation focused on the final answer, and the user can expand it to inspect the model's reasoning (same content the CLI prints above the answer). One block per turn; reset at tool boundaries and on `agent_end`. Clicking the `Thinking` summary to expand also refocuses the chat input on mouse toggle, so the next Cmd+V targets the textarea (HTML gives the `<summary>` focus on click, which would otherwise redirect paste elsewhere).
- `/pick-skill` opens a modal to toggle skills in the merged system prompt (RPC `get_skill_picker_state` / `set_skill_picker`), styled like `/pick-tools`; with no skills loaded, the chat shows an informational message instead of an empty dialog.
- `/tools` lists active tool names and optional MCP/extension groups (RPC `get_tool_picker_state`). `/pick-tools` opens a modal to toggle those groups (RPC `set_tool_picker`). **Note:** the embedded agent uses `--no-extensions` by default, so groups are often empty unless the spawn configuration loads extensions and MCP (same data as terminal `free-code` when those are loaded).
- Slash command palette: type `/` in the chat input to open a **Commands** / **Skills** menu (from RPC `get_commands` plus built-in `/session`); use arrows and Enter or click to insert. Sending `/session` still shows live session stats for the active tab.
- `/session` is handled in the extension host: it calls RPC `get_session_stats` (session JSONL on disk) and renders the summary in the chat **without** sending a prompt to the model (avoids relying on an up-to-date `free-code` build for `prompt` + `session_info`). The RPC `session_info` event still updates the same UI if emitted.
- Initial VS Code/Cursor extension MVP with sidebar chat webview backed by `free-code --mode rpc`.
- Setting `free-code.env` to pass environment variables (for example `GOOGLE_CLOUD_PROJECT` and `GOOGLE_CLOUD_LOCATION`) to the `free-code` process; the GUI does not inherit terminal `export`s.
- Chat history persisted per workspace (`workspaceState`), restored when the webview reloads; webview `retainContextWhenHidden` keeps the panel when hidden.
- **New chat** (`+` in the toolbar) aborts any in-flight turn, calls RPC `new_session`, and clears UI and stored history.
- **Busy** state disables send and new chat while a response is streaming.
- **Multiple conversations**: tab bar with one tab per session, **+** to add (RPC `new_session` + new `sessionFile`), click a tab to `switch_session` to that file, **×** to close; state stored in `workspaceState` under `freeCode.chatView.tabs.v1` (includes `sessionPath` per tab). Migrates legacy `freeCode.chatView.history` into a single tab.

### Changed

- Chat textarea: **Enter** submits the message; **Shift+Enter** inserts a newline (placeholder updated).
- **Drag and drop** files onto the input (e.g. from the Explorer): inserts `[file #N basename]` chips like the TUI, expands to shell-style path tokens on send (see `@free/pi-tui` editor file-drop behavior). Uses `text/uri-list` and Electron `File.path` when available.
- **Fixed** file drop/paste in the webview: `document` capture `dragover`/`drop` + broader `DataTransfer` handling (async `getAsString` for string items), paste path resolution in the same tick when possible. Fallback command **Free Code: Insert file paths in chat** (`free-code.attachFiles`) opens a file picker and inserts the same chips.
